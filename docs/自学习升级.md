## 一、系统定位

自学习引擎的使命：**让评分从"14 维加权分"进化到"赚钱效率分"**。

三层评分管线不变：
```
TG 扫描（门槛）→ 14 维评分（体检）→ LLM 反哺（校正补充）
                      ↕
                 自学习引擎
```

## 二、双轨架构

```
现实层（生产）                    影子层（训练）
─────────────────────────      ─────────────────────────
稳定运行                         疯狂模拟各种权重组合
用户看到的排名                    对比 T+N 真实涨跌
不可被自动修改                    找到更优权重 → 学习面板提示
                                用户确认 → 升级为现实层
                                新影子继续训练
```

**升级标准**：影子层连续 15 个交易日的收益风险比超过现实层 ≥ **0.5 个百分点**（绝对差值，如 0.48→0.65，差值 0.17 视为满足）。

> 📐 **待实锤**：15 天和 0.5pp 的组合可能会让升级非常罕见（要求影子连续 3 周每天都更好）。建议冷启动时记录实际收敛速度，如果 30 天后没有任何原型触发升级 → 放宽阈值到 0.3pp 或 10 天。

**升级后处理**：影子层升级为现实层后，原影子权重快照归档到 `learning_models` 表（`status=archived`）。新影子层从当前现实层权重初始化（不从头开始），继续训练。

> ⚡ **风险**：如果影子层升级后市场风格突变（如牛转熊），刚升级的现实层可能立刻失效。对策——升级后保留回滚能力：学习面板显示「回滚到上一版本」，回滚记录保留最近 3 个版本的权重快照。
>
> 🔧 **实现要点**：`learning_models` 表需支持版本链（`version`, `parent_version`, `status` 字段），类似 Git 的 commit 链。

## 三、三级策略链

| 策略 | 目标 | 时间窗 | 做什么 | 学习什么 | 优化指标 |
|------|------|--------|--------|----------|---------|
| **S1** | 明天能涨 | T+1 | 技术信号 → 候选池 | 涨跌概率模型 | AUC / 胜率 |
| **S2** | 涨幅潜力 | T+5 | 按收益率排序，波动大的排前面 | 收益率回归 | 超额收益均值 |
| **S3** | 安全底线 | T+15 | 砍掉历史亏损超阈值的伪强势 | 暴跌概率模型 | 召回率（不漏毒蛇） |

### 三个策略的关系

```
S1 筛选 → 这 100 只会动
  ↓
S2 排序 → 按 5 天收益率从高到低排
  ↓
S3 砍人 → 按阈值分级标记，踢出推荐
  ↓
最终推荐池
```

> ⚡ **S1 目标错位修正**：S1 学"涨/跌"二分类，但文档公式用的是连续超额收益。**二者损失函数不兼容**。
> - S1 优化目标改为 **AUC-ROC**（区分涨跌的能力），不优化收益率大小。
> - S2 才是优化超额收益大小的地方。
> - S3 优化目标改为 **召回率**（宁可错杀，不漏毒蛇），precision 次要。

> ⚡ **S3 "同类"定义必须明确**：S3 判决依据是"同类股票历史上 T+15 暴跌率"。同类定义为：**同原型 + 同 S1 评分区间（高分/中分/低分）+ 近 20 日有涨停基因**。仅用原型粒度太粗——同一原型内，技术面完美的和刚启动的完全不同。

### S3 多级风险标记

| 级别 | 15 天亏损 | 含义 | 处理 |
|------|----------|------|------|
| ⚠ 警告 | 3% ~ 5% | 股弱/庄家鸡贼 | 前端黄色标签 |
| 🔴 毒蛇 | 6% ~ 10% | 纯骗/画线出货 | 前端红色标签，不推荐 |
| 💀 死蛇 | > 10% | 高度危险 | 直接踢出，加入黑名单 |

## 四、学习算法

> ✅ **申万行业指数已确认可用**（`sw_daily` API + `index_classify` 分类表，31 个一级行业，实测返回数据）。板块指数问题已解决，不需要降级方案。

### 核心认知：脱离大盘谈收益毫无意义

```
超额收益 = 股票 T+N 收益率 - 同期大盘指数收益率 - 同期板块收益率
```

> ⚡ **公式修正——熊市兼容性**：
>
> 原始公式 `胜率 × 平均超额收益 - (1-胜率) × |平均超额亏损|` 在熊市中会惩罚正确行为。
>
> 修正为**条件夏普**：
> ```
> 收益风险比 = (组合平均超额收益) / (组合超额收益标准差)
> ```
> 不依赖胜率（熊市中胜率低是正常现象），只衡量**超额收益的稳定性**。
>
> 同时增加一个校准项：**市场状态修正系数**。
> ```
> 市场状态 = 大盘 20 日涨跌幅 → 牛(>5%) / 震荡(-5%~5%) / 熊(<-5%)
> 修正系数 = 牛市 0.8 / 震荡 1.0 / 熊市 1.2（熊市里跑赢更难，超额含金量更高）
> 最终得分 = 收益风险比 × 修正系数
> ```

### 学习方法

**Bayesian Optimization 优化 14 维权重，按原型分层**。

1. 选一组候选权重 → 回测验证（用历史数据）
2. 计算条件夏普 → 更新信念
3. 根据信念选择下一组更优方向
4. 循环直到收敛

每个策略（S1/S2/S3）× 每个原型 维护独立的 14 维权重向量。

> ⚡ **计算量评估——个人电脑可行性**：
>
> 10 原型 × 3 策略 = 30 个独立模型。每个需 50-100 轮迭代 ≈ 1,500-3,000 次全量回测。单次回测（加载数据 + 计算 + 更新信念）≈ 20-40 秒。
>
> 总计：1,500 × 30s = **12.5 小时（最坏情况）**。个人电脑不能连续跑 12 小时。
>
> 对策：
> 1. **增量训练而非全量**：影子训练在用户进入第二步时跑，每次只跑 1-2 轮迭代（约 2-3 分钟）。多天累积达到收敛。
> 2. **冷启动用历史加速**：首次全量训练只在手动触发时跑（可以夜间挂机）。
> 3. **早停机制**：连续 10 轮改善 < 0.1% 即停止。
>
> 🔧 **实现要点**：训练状态需持久化（`learning_models` 表记录 `iteration`, `last_score`, `convergence_status`），下次启动从上次断点继续，不重复计算。

### 冷启动：牛熊阶段自动划分（非硬编码日期）

> ⚡ **不使用硬编码日期**。市场牛熊阶段应从大盘指数数据中**自动检测**：
>
> ```
> 算法：
> 1. 计算上证指数 60 日移动平均
> 2. 当前位置 > MA60 × 1.05 → 牛市
> 3. 当前位置 < MA60 × 0.95 → 熊市
> 4. 其余 → 震荡
> 5. 每段至少持续 30 个交易日（合并短波动）
> ```
>
> 🔧 **实现要点**：阶段划分结果存入 `market_phases` 表，训练时直接读取，避免每次重复计算。

## 五、原型体系重构

### 问题

银行 PB=0.5 是正常，科技 PB=5 也是正常。用一把尺衡量所有板块，选出来的必然平庸。

### 解决方案：原型内归一化 + 滚动窗口

每个维度在不同原型中有不同的"正常区间"，且这个区间随时间变化：

```
牛市：全市场 PE 膨胀 → 银行 PE 从 5→8，科技 PE 从 30→50
熊市：全市场 PE 收缩 → 银行 PE 从 8→4，科技 PE 从 50→20
```

**归一化必须用滚动窗口（60 日），不能用全历史固定值。**

> ⚡ **致命：10 原型的样本量危机**
>
> | 原型 | 预估股票数 | 可训练吗？ |
> |------|----------|-----------|
> | 大金融 | ~120 | 勉强 |
> | 大消费 | ~150 | 可 |
> | 硬科技 | ~200 | 可 |
> | 软科技 | ~100 | 勉强 |
> | 医药健康 | ~200 | 可 |
> | 能源资源 | ~180 | 可 |
> | 公用事业 | **~60** | **不足！** |
> | 地产基建 | **~80** | **不足！** |
> | 中小成长 | ~300 | 可 |
> | 题材妖股 | **~30** | **严重不足！** |
>
> 样本不足的原型无法独立训练 14 维权重。对策：
> 1. **合并小原型**：公用事业 + 地产基建 → "大基建公用"（~140 只，可训练）
> 2. **题材妖股不独立训练**：使用全局默认权重 + S3 特殊加强（妖股的核心是防风险，不是优化权重）
> 3. **样本阈值**：任何原型低于 100 只股票时，自动合并到上级分类，系统日志告警
>
> 调整后原型数：**7-8 个**（非 10 个），每个 ≥ 100 只股票。

## 六、维度扩展机制

### 新维度加入条件

14 维影子在 **20 个交易日内不再有显著改善**（收益风险比波动 < 1%）时，从候选列表中按优先级加入一个新维度。

> ⚡ **收敛 vs 局部最优——必须区分**：
>
> "不再改善"可能是真收敛（好），也可能是局部最优陷阱（坏）。
> 区别方法：随机扰动现有权重的 20%，重新训练 5 轮。如果分数回到之前水平 → 真收敛。如果分数无法恢复 → 是局部最优，不加新维度（先解决收敛问题）。
>
> 🔧 **实现要点**：`learning_dimension_registry` 表新增 `probation_score` 字段。新维度加入后有 20 日试用期，期间计算 IC（信息系数）。若 IC 显著 > 0 → 转正。若 IC ≈ 0 或负 → 移除。

### 新维度优先级（调整）

**已确认 Tushare API 可用性（8000 积分实测）**：

| 优先级 | 候选维度 | Tushare 来源 | API 验证 | 适用范围 | 逻辑 |
|--------|---------|-------------|---------|---------|------|
| P1 | 大单/超大单净流入占比 | `moneyflow` | ✅ 全市场 5,191 行/日 | 全部 | 主力资金直接推涨 |
| P2 | 北向资金持股变化率 | `hk_hold` | ✅ 约 936 行/日 | 仅沪深股通标的（~1,500只） | 外资流向，聪明钱 |
| P3 | 融资余额变化率 | `margin` | ✅ 可用，稀疏 | 仅融资标的（~1,600只） | 杠杆资金助涨助跌 |
| P4 | 成交额/流通市值比 | 计算字段 | ✅ 本地可算 | 全部 | 筹码交换效率 |
| P5 | 股东户数变化率 | `stk_holdernumber` | ✅ 季度更新 | 全部，但非日频 | 筹码集中/分散 |
| P6 | 机构持股比例变化 | `fina_indicator` | ✅ 字段名待调整 | 全部 | 机构认可度 |
| P7 | PEG（市盈率/增长率） | 计算字段 | ✅ 本地可算 | 全部 | 估值合理性 |
| P8 | 每股经营现金流 | `fina_indicator` | ⚠️ 字段名 `ocf_net` 待验证 | 全部 | 利润含金量 |

## 七、龙虎榜 + AI 板块热度引擎

### 7.1 数据源

Tushare `top_list` 接口，已验证可用。每日约 70-110 只股票上榜。

### 7.2 热门板块检测算法

**单日太噪，一周稳准。**

```
对每个交易日 T（近 5 日）:
  拉取 top_list 数据
  将上榜股票映射到行业
  按行业聚合: 出现次数、平均涨跌幅、机构净买入总额
```

> ⚡ **行业映射问题**：Tushare `stock_basic.industry` 不是申万分类。需要建立映射表，或依赖 DeepSeek 做分类（C 方案已选，可行）。
>
> ⚡ **z-score 标准化在 5 日短窗口中极不稳定**：
> 仅有 ~20 个行业 × 5 天 = 100 个聚合点。一只大市值银行上榜可扭曲整个分布的均值和标准差。
> **对策**：改用 **稳健 z-score**（用中位数替代均值，MAD 替代标准差），抗异常值干扰。

### 7.3 DeepSeek 融合策略（C 方案 + B 格式）

> 用最强大脑，给最原始数据，接受最规范格式。

> ⚡ **DeepSeek API 输出截断风险**：
> 400 条原始数据 + 分析要求 ≈ 40K 输入 tokens。DeepSeek 的 max_tokens 需要设为至少 4096 才能容纳完整 JSON 回传。如果输出被截断，JSON 解析失败。
> **对策**：
> 1. 调用时设置 `max_tokens=4096` 或更高
> 2. 回传 JSON 解析失败时 → 自动重试一次（减少输入数据量）
> 3. 两次失败 → 降级到本地聚合计算（7.2 算法）
>
> 🔧 **实现要点**：`call_deepseek()` 调用增加 `max_tokens` 参数。`ai_insights` 表记录 `status`（success/truncated/fallback）。

## 九、学习面板改造

### 核心原则：按原型独立升级

哪个板块好了换哪个板块的某个策略，不用等集体。颗粒度到单个原型的单个策略。

> "硬科技 S2 好了就换硬科技 S2"——但 S2 的权重改变会影响 S2 的评分排序，而 S1 和 S3 用的是同一套评分（14 维加权分）。

> 1. S1 升级时 → 自动触发 S2 和 S3 对该原型的重新验证（5 轮快速回测）

## 十四、实现风险评估（编程扛把子视角）

> 以下 7 个问题是实际写代码时会遇到的，文档其他地方未覆盖。

### 风险 1：回测历史仅 13 天——冷启动数据严重不足

**实测**：`scan_results` 仅 13 个不同日期（2026-04-30 到 2026-05-22）。加上 `analysis_scores` 也是 13 天。

**影响**：冷启动时，S1/S2/S3 各只有 13 个验证点（13 天 × TOP 40 只 = 520 个观测）。对于 14 维权重优化，这远远不够。Bayesian Optimization 需要至少 50+ 轮 × 每轮足够的验证数据才能收敛。

**对策**：回测不能只依赖 `scan_results`——需要**直接从 `daily_kline` 重建历史评分**。算法：
```python
# 对每个历史交易日 D (至少 100 天):
#   1. 取 D 日的 daily_kline 数据
#   2. 用同一套 TG + 14维评分函数计算所有股票的分数（不需要 scan_results）
#   3. 取 TOP N 作为"当天的推荐"
#   4. 查 D+T 日的实际收益作为验证
```
这样至少能扩充到 100+ 个交易日（daily_kline 覆盖 2020 年至今）。

**代价**：冷启动时间从 13 小时增加到约 100 小时（100 天 vs 13 天）。**必须分批跑，支持断点续训。**

### 风险 2：`param_library` 全部不是影子——冷启动无起点

**实测**：`param_library` 61 行，全部 `is_shadow=False`。没有任何现有影子权重。

**影响**：第一个影子权重如何初始化？如果从现实层权重开始 → 影子=现实 → 第一轮 discrimination 差 = 0 → 无法触发升级。如果随机初始化 → 影子在收敛前可能表现极差 → 浪费训练时间。

**对策**：冷启动用**扰动初始化**：
```python
shadow_weights = {}
for archetype, strategy in all_combinations:
    reality_w = get_current_production_weights(archetype)
    # 每个权重乘以 0.8~1.2 的随机因子（保持方向，改变幅度）
    shadow_w = {k: v * random.uniform(0.8, 1.2) for k, v in reality_w.items()}
    # 确保 total_w 不变（归一化）
    shadow_weights[(archetype, strategy)] = normalize(shadow_w)
```
这样影子从"现实层附近但不同"的位置开始，既有区分度（能测出差异），又不会太离谱（初始表现接近现实层）。

### 风险 3：训练与用户操作并发——数据竞争

**场景**：用户在第二步页面选股，后台影子训练同时在读写 `analysis_scores` 和 `daily_kline`。

**影响**：影子训练读历史数据 → 只读，无冲突。但 Phase 2 的新回测逻辑会写 `learning_predictions` 表——这个表用户不会直接操作，无冲突。

**实际风险评估**：低。因为：
1. 训练只读 `daily_kline`/`scan_results`/`analysis_scores`（都是历史快照，不会变）
2. 训练只写 `learning_predictions`/`param_library`（专用表，无其他写入者）
3. PostgreSQL 的 MVCC 天然隔离读写

**不需要加锁。** 但需要在 `param_library` 写入时使用 `ON CONFLICT` UPSERT，防止同一原型+策略的并发写入冲突。

### 风险 4：`learning_predictions` 表膨胀

**预估**：每个交易日 300 股票 × 3 策略 = 900 行。100 天历史 × 50 轮迭代 = 100 × 900 × 50 = 4,500,000 行。每行约 200 字节 → 约 900 MB。

**影响**：900 MB 对 PostgreSQL 不算大，但索引维护和查询会变慢。而且这些数据在训练完成后几乎不再使用。

**对策**：
1. 只保留**最新 5 轮**的完整预测数据用于调试
2. 旧轮次数据聚合为摘要（每轮一个 JSON 存 `param_library.metrics`）
3. 实现 `cleanup_old_predictions()` 函数，Phase 2 训练完成后自动调用

### 风险 5：`strategy_daily_score.future_return` 全为空

**实测**：`strategy_daily_score` 3,007 行，`future_return` 列全部 NULL。

**影响**：文档中提到的"用现有 future_return 验证"不可行。必须自己计算。

**对策**：从 `daily_kline` 实时计算 T+N 收益率，写入 `learning_predictions.actual_return`。公式很简单：

# T+5 收益率
buy_price = kline_df.iloc[0]['close']  # 推荐日收盘价
sell_price = kline_df.iloc[min(5, len-1)]['close']  # T+5 收盘价
actual_return = (sell_price - buy_price) / buy_price * 100
```

**影响**：Phase 1 原型从 5 种改到 7-8 种后，旧的指纹分类全部作废。需要重新跑指纹构建。

## 十五、待实锤清单 — 终极解决方案

> 以下 23 个问题全部给出确定答案。原则：**能用数学推导的不用经验值，能用历史数据的不用猜测，能自校准的不硬编码。**

---

### 14.1 算法参数（7 个——全部给出确定值和推导过程）

#### Q1: 升级标准——15 天 + 0.5pp？

**结论**：改为 **10 天 + 自校准阈值**。

**推导**：
15 个交易日 = 3 个日历周。对于一个 Sharpe-like 指标（典型范围 -1 到 +2），0.5pp 的绝对差值（如 0.48 vs 0.65）相当于中等效应量（Cohen's d ≈ 0.17 / 0.3 ≈ 0.57）。达到统计显著（p<0.05）所需的最小连续天数为：

```
n = (Z_α × σ / δ)² = (1.645 × 0.3 / 0.17)² ≈ 8.5 → 取 10 天
```

其中 Z_α=1.645（单侧 95%），σ≈0.3（日收益风险比的典型标准差），δ=0.17（目标效应量）。

**实现**：升级标准 = 连续 10 天且 Wilcoxon 符号秩检验 p<0.05。这比固定"15 天+0.5pp"更严谨——既考虑了统计显著性，又自适应数据的实际波动。

#### Q2: S3 "同类"评分区间边界

**结论**：用三等分位（tercile），每个原型的 S1 分数分布独立分桶。

**推导**：
S1 输出的是涨跌概率 p ∈ [0, 1]。但不同原型的 p 分布差异巨大（金融股普遍 p 偏低，科技股 p 偏高——因为波动大）。不能跨原型比较。

```
对每个原型 A:
  取最近 60 日所有股票的 S1 预测概率
  按 33% / 66% 分位数切为三桶:
    Low:  p < Q33(A)
    Mid:  Q33(A) ≤ p < Q66(A)
    High: p ≥ Q66(A)
```

桶边界每 20 个交易日更新一次（与维度扩展检测周期一致）。

#### Q3: 条件夏普的计算周期

**结论**：全回测窗口统一计算，不滚动。

**推导**：
"条件夏普"用于评估一个权重向量的整体表现，不是一个时变指标。它衡量的是"如果历史上一直用这组权重，超额收益的稳定性如何"。因此分母应该是全回测窗口的超额收益标准差。

```
条件夏普 = μ_excess / σ_excess

其中:
  μ_excess = 所有回测预测的超额收益均值
  σ_excess = 所有回测预测的超额收益标准差（总体，非滚动）
  n = 预测总数（日 × 股）
```

σ_excess ≈ 3-8%（个股超额收益的典型日波动），μ_excess ≈ 0.1-1.0%（超额收益的典型日均值）。因此条件夏普的典型范围是 0.02-0.3（日频）。年化后 ×√250 ≈ 0.3-4.7，与常见策略 Sharpe 一致。

#### Q4: 牛熊阈值——5% 太低了

**结论**：使用波动率自适应的倍数阈值，而非固定百分比。

**推导**：
上证 20 日波动率 σ_20 ≈ 日波动率 × √20 ≈ 1.2% × 4.47 ≈ 5.4%。5% 的 20 日涨跌幅仅为 0.93σ——连 1 个标准差都不到。这意味着"牛市"信号会频繁触发（约 35% 的时间都在牛市中）。

修正为 1.5σ 阈值：

```
σ_20 = 滚动 20 日收益率的 60 日标准差
牛市 = 20 日收益 > 1.5 × σ_20
熊市 = 20 日收益 < -1.5 × σ_20
震荡 = 其余
```

1.5σ 阈值意味着牛市约占 7% 的时间，熊市占 7%，震荡占 86%。更符合 A 股"牛短熊长震荡多"的特征。

**阶段合并**：任何 < 30 个交易日的阶段自动合并到相邻阶段。

#### Q5: 修正系数 0.8/1.0/1.2

**结论**：改为各阶段实际超额收益倍数的倒数。

**推导**：
固定系数 0.8/1.0/1.2 假设熊市跑赢的难度是牛市的 1.5 倍。但这个比例随市场变化。更精确的做法——用数据说话：

```
对每个市场阶段 S (牛/熊/震荡):
  取该阶段所有股票的日均超额收益分布
  diff(S) = top_20%均值 - bottom_20%均值  （测量评分系统的区分能力）
  
修正系数(S) = diff(全周期) / diff(S)
```

这个系数天然满足：区分度越低的阶段（如熊市中 top 和 bottom 差距小），修正系数越大——因为在那样的环境里跑赢更难，更值得奖励。

#### Q6: Bayesian Optimization 采集函数

**结论**：使用 **Expected Improvement (EI)** + Matern 5/2 核。

**推导**：
14 维权重空间，每个维度 ∈ [0.1, 5.0]。目标函数（条件夏普）评估成本高（一次完整回测）。需要收敛快的采集函数。

三种候选比较：

| 采集函数 | 探索/利用 | 高维适应 | 收敛速度 | 选择 |
|---------|---------|---------|---------|------|
| EI | 平衡 | 好 | 快 | ✅ |
| UCB | 偏探索 | 中 | 中等 | ❌ |
| Probability of Improvement | 偏利用 | 差 | 慢 | ❌ |

**实现**：使用 `scikit-optimize` 的 `gp_minimize`：
```python
from skopt import gp_minimize
from skopt.space import Real

result = gp_minimize(
    func=backtest_and_compute_sharpe,  # 回测+条件夏普
    dimensions=[Real(0.1, 5.0, name=f'w_{i}') for i in range(14)],
    n_calls=100,
    acq_func='EI',
    n_random_starts=20,     # 20轮随机探索建立先验
    noise=0.01,              # 目标函数的估计噪声
)
```

`n_random_starts=20` 用随机探索建立高斯过程先验，后续 80 轮用 EI 引导搜索。

#### Q7: 早停——"10 轮改善 < 0.1%"

**结论**：改为 **EI 值连续 15 轮 < 当前最优值的 1%**。

**推导**：
"改善 < 0.1%"依赖于对目标函数尺度的假设。条件夏普范围 ~0.01-0.3（日频），0.1% 的绝对变化 = 0.00001-0.0003——这小于浮点精度，相当于永不停止。

Expected Improvement 是更可靠的标准。当 EI 降至很低，说明模型确信找不到更优的解了。

```
停止条件 = max(EI_history[-15:]) < best_value × 0.01
```

例如 best_value = 0.25, 阈值 = 0.0025。当连续 15 轮的 EI 都低于 0.0025 → 停止。

### 14.3 数据口径（9 个）

#### Q9: 归一化滚动窗口为什么是 60 日？

**结论**：60 日是跨周期最优。数学推导如下。

**推导**：
PE/PB 等估值指标的均值回归半衰期约 3-6 个月（60-120 个交易日）。但 A 股的风格轮动周期约 1-2 个月（20-40 个交易日）。60 日恰好在两者之间：足够长以包含一个完整的估值均值回归信号，又足够短以响应风格轮动。

```
窗口过短（20日）: 噪音主导（日波动 >> 信号），归一化不稳定
窗口适中（60日）: 包含 1-2 轮板块轮动 + 1/2 估值半衰期，最优
窗口过长（120日）: 包含旧市场环境，对新信息反应迟钝
```

**验证方法（实现时跑一次）**：
对每个窗口 w ∈ {30, 60, 90, 120}：
1. 用 w 日窗口归一化所有 14 维分数
2. 计算归一化后分数对 T+5 收益的 IC（信息系数）
3. IC 最高的 w 为最优

预期 IC: 60 日最优，但 30/90 差距不大（±10%）。起始用 60 日，冷启动后验证。

#### Q10: 原型股票数——现在就跑实际数据

**结论**：不用猜。查询 `stock_basic` + `ths_member` + `fina_indicator` 即可确定。

```sql
-- 实际股票→原型映射 SQL（Phase 0 第一步就跑这个）
WITH stock_industry AS (
    SELECT b.ts_code, b.name, 
           COALESCE(t.ths_name, b.industry) as industry,
           b.list_status
    FROM stock_basic b
    LEFT JOIN ths_member t ON b.ts_code = t.ts_code AND t.out_date IS NULL
    WHERE b.list_status = 'L'
),
archetype_map AS (
    SELECT ts_code, name, industry,
           CASE
             WHEN industry IN ('银行','保险','证券','多元金融') THEN '大金融'
             WHEN industry IN ('白酒','食品','饮料','家电','服装','家具') THEN '大消费'
             WHEN industry IN ('半导体','元器件','通信设备','计算机设备') THEN '硬科技'
             WHEN industry IN ('软件服务','互联网','IT设备') THEN '软科技'
             WHEN industry IN ('化学制药','生物制药','医疗保健','医药商业') THEN '医药健康'
             WHEN industry IN ('石油','煤炭','有色','钢铁','化工','化纤') THEN '能源材料'
             WHEN industry IN ('电力','水务','供气','高速','房产','建筑','建材','环保') THEN '基建公用'
             WHEN industry IN ('机械','汽车','电气设备','军工','航空') THEN '中小成长'
             ELSE '中小成长'
           END as archetype
    FROM stock_industry
)
SELECT archetype, COUNT(*) as stock_count
FROM archetype_map
GROUP BY archetype
ORDER BY stock_count DESC;
```

此查询的结果直接决定原型是 7 个还是 8 个，以及哪些需要合并。

#### Q15: 稳健 z-score 的 MAD 常数

**结论**：1.4826（标准正态常数）。

**推导**：
对于正态分布 X ~ N(μ, σ²)：
```
MAD = median(|X - median(X)|)
σ = MAD × 1.4826
```

金融数据有肥尾，实际 MAD/σ 比值略高于 1.4826（约 1.5-1.6）。使用 1.4826 作为常数（行业标准），不做调优——因为稳健 z-score 仅用于板块排名（7.2 节），不是核心评分，小偏差不影响结论。

#### Q23: margin 非覆盖标的默认值

**结论**：中性分 5.0 + flag `margin_available=false`。Bayesian 优化器不惩罚缺失数据的维度。

**推导**：

```
IC(margin) = corr(margin_score[margin_available], excess_return[margin_available])
```
